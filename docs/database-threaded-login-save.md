# Per-Thread Database, Async Login, and Player Save Replay

This fork now follows the Canary database concurrency model in the places that matter for mass login and global saves.

## Database connections

`Database` no longer uses a single shared `MYSQL*` guarded by one global mutex. Each thread that touches the database lazily opens its own `ConnectionContext`, using the credentials captured during startup. A transaction stays on the calling thread, so `BEGIN`, write queries, `COMMIT`, `ROLLBACK`, and `getLastInsertId()` keep connection affinity.

The MySQL client library is initialized once at startup. Threads that open a MySQL connection call `mysql_thread_init()`, and a thread-local cleanup object calls `mysql_thread_end()` when that thread exits.

## Async login

`ProtocolGame::login()` still runs validation on the dispatcher. Before the heavy player load, it reserves the character GUID with `Game::reserveLogin()` so duplicate concurrent logins are rejected. The worker thread loads the player while the player is not yet placed in the world. `ProtocolGame::finishLogin()` returns to the dispatcher, loads deferred world data such as guild state, places the creature, restores client flags, and releases the login reservation on every exit path.

## Player save replay

`IOLoginData::buildPlayerSave()` runs on the thread that owns the player state and captures SQL generated by `savePlayerQueries()` instead of executing it. `IOLoginData::flushPlayerSave()` replays the captured SQL inside a transaction on a worker thread through `DBTransaction::executeWithinTransactionRollbackOnFailure`. That helper reapplies the full transaction up to 3 times when `Database::lastQueryWasDeadlock()` reports a deadlock or lock timeout, including rollback/commit retry handling, so player flushes are resilient to transient database contention.

`SaveManager::saveAll()` builds player saves on the dispatcher and flushes them on the thread pool. Per-player flush ordering is guarded by `flushInFlight` and `pendingFlushes` so two saves for the same GUID are never flushed concurrently. Each player save carries a storage dirty-key snapshot; after a successful flush, the dispatcher only acknowledges the keys from that snapshot whose dirty revision has not changed, leaving later storage changes dirty for the next save.
